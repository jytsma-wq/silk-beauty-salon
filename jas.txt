Task: Comprehensive Route Connectivity Audit for Silk Beauty Salon Next.js Application

Context:
- Next.js 16 App Router application
- Multilingual routing with next-intl (6 languages: en, ka, ru, tr, ar, he)
- Route structure: /[locale]/[...pages]
- Files located in: /src/app/[locale]/

Objective:
Perform a complete audit of all routes, links, navigation elements, and redirects to ensure every path is properly connected, functional, and error-free.

Instructions:

STEP 1: ROUTE DISCOVERY
1. Scan /src/app/[locale]/ directory recursively
2. List all route segments (folders) and page.tsx files
3. Identify all dynamic routes ([id], [...slug], etc.)
4. Document the complete route tree structure
5. Check for any missing page.tsx, layout.tsx, or error.tsx files
6. Verify all route groups (parentheses folders) are properly configured

STEP 2: LINK EXTRACTION
1. Search all .tsx and .ts files for navigation elements:
   - <Link> components from 'next/link'
   - <a> tags with href attributes
   - useRouter() and router.push() calls
   - redirect() function calls
   - Router.push() calls
   - Any navigation in middleware.ts
2. Extract every unique href/path value
3. Categorize links:
   - Internal routes (starting with /)
   - External URLs (http/https)
   - Hash links (#)
   - Relative links
   - Dynamic links with variables

STEP 3: ROUTE VALIDATION
For each discovered link:
1. Check if the target route exists in the file structure
2. Verify locale prefix is correctly applied
3. Check if dynamic parameters are properly handled
4. Validate anchor links point to existing IDs
5. Ensure API routes (/api/*) exist
6. Check for typos in route paths
7. Verify all treatment slugs in links match database/data

STEP 4: NAVIGATION COMPONENTS AUDIT

1. /src/components/layout/Header.tsx - all nav links

4. Language switcher - all locale switching links

STEP 5: MIDDLEWARE & REDIRECTS
1. Check /middleware.ts:
   - Locale detection logic
   - Redirect rules

2. Check next.config.ts:
   - Redirects array
   - Rewrites array
3. Verify 404 handling with not-found.tsx files

STEP 6: INTERNATIONALIZATION ROUTES
For each of the 6 locales (en, ka, ru, tr, ar, he):
1. Verify all routes are accessible with locale prefix
2. Check translation files have keys for all route names
3. Verify RTL routes (ar, he) load correctly
4. Check locale switcher redirects to correct pages
5. Validate alternate language links (hreflang)

STEP 7: API ROUTES
1. List all API routes in /src/app/api/
2. Check all fetch() calls point to existing API routes
3. Verify API route methods (GET, POST, etc.) match usage
4. Check for any hardcoded API URLs that should be relative

STEP 8: SPECIAL CASES

2. Verify contact form submission endpoints
3. Check newsletter signup endpoints
4. Verify sitemap.xml generation includes all routes
5. Check robots.txt allows all necessary routes
6. Verify any auth-protected routes (if applicable)

STEP 9: BROKEN LINK DETECTION
Create a report of:
1. Dead links (pointing to non-existent routes)
2. Orphaned pages (routes that nothing links to)
3. Circular redirects
4. Missing translation keys for route names
5. Inconsistent route naming (kebab-case violations)
6. Links missing locale prefix
