### 📌 PROMPT 4: Robust Error Handling & Monitoring

```
Task: Implement comprehensive error handling, logging, and monitoring system

Context:
- Need production-grade error handling
- Implement centralized logging
- Add real-time error monitoring
- Set up alerting for critical issues

Requirements:

1. **Error Boundaries**
   - Create global error boundary component
   - Add route-specific error boundaries
   - Implement error recovery strategies
   - Add user-friendly error messages in all 6 languages
   - Create error fallback UI components
   - Add retry mechanisms

2. **API Error Handling**
   - Create standardized API error response format
   - Implement error classification (4xx vs 5xx)
   - Add request ID tracking
   - Implement retry logic with exponential backoff
   - Add circuit breaker pattern for external APIs
   - Create error mapping for internationalization

3. **Error Logging**
   - Integrate Sentry or similar service
   - Add structured logging with Winston or Pino
   - Implement log levels (error, warn, info, debug)
   - Add request/response logging middleware
   - Log security events separately
   - Add performance logging
   - Implement log rotation
   - Add PII redaction in logs

4. **Frontend Error Tracking**
   - Track JavaScript errors
   - Track unhandled promise rejections
   - Track resource loading errors
   - Track API call failures
   - Add user session tracking
   - Track custom events
   - Add source map support for debugging

5. **Backend Error Tracking**
   - Track API route errors
   - Track database errors
   - Track external API failures
   - Track middleware errors
   - Track cron job errors (if any)

6. **Monitoring & Alerting**
   - Set up uptime monitoring (Pingdom/UptimeRobot)
   - Add health check endpoints
   - Implement status page
   - Set up PagerDuty/Opsgenie alerts
   - Add Slack notifications for critical errors
   - Monitor error rates and trends
   - Set up custom dashboards

7. **Graceful Degradation**
   - Implement fallback strategies for API failures
   - Add offline mode support
   - Cache critical data locally
   - Show meaningful error messages
   - Provide alternative actions

8. **Error Recovery**
   - Add automatic retry mechanisms
   - Implement queue for failed operations
   - Add manual retry UI components
   - Clear cache on certain errors
   - Redirect to safe pages on critical errors

Deliverables:
- Global error boundary in /src/components/error-boundary.tsx
- API error handling middleware in /src/lib/api-error-handler.ts
- Centralized logger in /src/lib/logger.ts
- Sentry configuration in /src/lib/sentry.ts
- Health check routes in /src/app/api/health/
- Error monitoring dashboard setup
- Alert configuration
- Error message translations in all language files
- Documentation in docs/ERROR_HANDLING.md

Error Handling Standards:
- All errors must be logged with context
- User-facing errors must be translated
- Critical errors must trigger alerts
- All API errors must include request ID
- Production errors must not leak sensitive data
```
